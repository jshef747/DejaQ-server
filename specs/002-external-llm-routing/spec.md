# Feature Specification: External LLM API Integration for Hard Prompt Routing

**Feature Branch**: `002-external-llm-routing`
**Created**: 2026-03-28
**Status**: Draft
**Input**: User description: "i want to add the api to the external llm that hard prompts will go there"

## User Scenarios & Testing *(mandatory)*

### User Story 1 - Hard Prompts Receive High-Quality Responses (Priority: P1)

When a user submits a complex or difficult query, the system automatically detects it as "hard" and routes it to an external, more capable language model. The user receives a response without needing to manually select which model to use.

**Why this priority**: This is the core value of the feature — ensuring that complex queries are answered with sufficient quality. Without this, hard prompts either fail or return low-quality local responses.

**Independent Test**: Can be fully tested by submitting a query classified as "hard" and verifying the response comes from the external model and contains a substantive answer.

**Acceptance Scenarios**:

1. **Given** a user submits a complex multi-step question, **When** the classifier labels it "hard", **Then** the response is generated by the external LLM (not the local model)
2. **Given** a hard prompt is routed externally, **When** the external service returns a response, **Then** the user receives the full response through the same chat interface with no behavioral difference compared to local responses
3. **Given** a user submits a "hard" prompt, **When** the external service is available, **Then** the response is returned within an acceptable time frame (under 30 seconds)

---

### User Story 2 - Easy Prompts Continue Using Local Model (Priority: P2)

When a user submits a simple query classified as "easy", it continues to be handled by the local model — the external LLM is not called, preserving cost efficiency.

**Why this priority**: A core design principle of the platform is cost reduction. External LLM calls cost money, so they should only be used when necessary. This story protects that invariant.

**Independent Test**: Submit an "easy" prompt and confirm it is handled locally without any external API call being made.

**Acceptance Scenarios**:

1. **Given** a user submits a simple factual question, **When** the classifier labels it "easy", **Then** only the local model is called and no external API request is made
2. **Given** the external LLM is unavailable or misconfigured, **When** an "easy" prompt is submitted, **Then** the system still responds normally using the local model

---

### User Story 3 - Graceful Failure When External LLM Is Unavailable (Priority: P3)

When the external LLM service is unreachable or returns an error, the system fails gracefully and informs the user rather than crashing or hanging indefinitely.

**Why this priority**: Reliability matters. External services can go down or have quota limits. Users should understand why their query did not complete rather than experiencing an unexplained failure.

**Independent Test**: Simulate an external service failure and confirm the user receives a clear error message and the system remains functional for subsequent requests.

**Acceptance Scenarios**:

1. **Given** a hard prompt is submitted, **When** the external LLM service returns an error or is unreachable, **Then** the user receives a clear message indicating the request could not be completed
2. **Given** the external LLM fails for one request, **When** a subsequent request is submitted, **Then** the system processes it normally without being in a broken state

---

### Edge Cases

- What happens when the classifier is uncertain (borderline hard/easy) and routes a prompt externally but the response is low quality?
- How does the system handle very long responses from the external LLM that exceed expected token limits?
- What happens if the external LLM API key is missing or invalid at startup?
- How does the system behave if the external LLM request times out mid-response?
- What happens when the external LLM is rate-limited?

## Requirements *(mandatory)*

### Functional Requirements

- **FR-001**: System MUST route queries classified as "hard" to the configured external LLM provider instead of the local model
- **FR-002**: System MUST route queries classified as "easy" exclusively to the local model, with no external API calls made
- **FR-003**: System MUST return the external LLM's response to the user through the existing chat interface (HTTP and WebSocket) without changing the response format
- **FR-004**: System MUST include conversation history when sending a hard query to the external LLM, maintaining multi-turn context
- **FR-005**: System MUST handle external LLM errors (network failure, rate limit, invalid key, timeout) gracefully and return a user-facing error message
- **FR-006**: System MUST support configuration of the external LLM provider and credentials via environment variables, without requiring code changes
- **FR-007**: System MUST log all external LLM routing decisions and response statuses, including errors, without logging sensitive user content
- **FR-008**: Cache miss responses from the external LLM MUST be eligible for the same generalize-and-store background pipeline as local LLM responses

### Key Entities

- **ExternalLLMRequest**: A request routed to the external provider, containing the user query, conversation history, and routing metadata (classifier label, confidence score)
- **ExternalLLMResponse**: The response returned by the external provider, including the generated text and any provider-level metadata (token usage, model name)
- **RoutingDecision**: The classifier output that determines whether a query goes to local or external LLM, with the difficulty label ("easy"/"hard") and confidence score

## Success Criteria *(mandatory)*

### Measurable Outcomes

- **SC-001**: 100% of queries classified as "hard" are routed to the external LLM when the service is available and configured
- **SC-002**: 0% of queries classified as "easy" trigger an external LLM call
- **SC-003**: Users receive a response (success or error message) for every submitted query — no silent failures or indefinite hangs
- **SC-004**: External LLM errors are surfaced to the user within 30 seconds of the failure occurring
- **SC-005**: The system remains fully operational for easy queries even when the external LLM is unavailable

## Assumptions

- The external LLM provider exposes a standard chat completion interface (request/response pattern similar to industry-standard AI APIs)
- A single external LLM provider will be integrated in this feature; support for multiple providers is out of scope
- The difficulty classifier (already implemented) reliably produces "easy"/"hard" labels that drive routing — no changes to classifier logic are in scope
- Streaming responses from the external LLM are out of scope for this iteration; full responses will be awaited before returning to the user
- The external LLM provider requires an API key for authentication, configured via environment variable
- Cost tracking or token usage reporting to end users is out of scope for this feature
- The existing cache pipeline (generalize → store in ChromaDB) applies equally to responses from the external LLM
